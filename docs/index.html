경로만 주면 자라나는 위키 — LLMwiki 워크플로우

물류플랫폼팀 AI 스터디 · 발표자 jenny.by · 클론 없이, 프롬프트 2개로 본인 업무용 위키를 만들고 키웁니다.

00왜 필요한가

AI에게 업무를 시켜본 사람은 안다. 매번 처음부터 설명해야 한다. "우리 팀은 미들마일이고, 이 기획서는 v1.6이고…" 세션이 끝나면 그 맥락은 전부 사라진다.

컨텍스트 소실 문제 — LLM은 대화가 끝나면 기억하지 못한다. 그래서 매 세션 같은 설명을 반복하고, 결과물의 포맷·깊이가 들쭉날쭉해진다.

LLMwiki는 이걸 파일로 푼다. 업무 컨텍스트와 작동 규칙을 CLAUDE.md에 한 번 적어두면, Claude Code가 매 세션 그 파일부터 읽고 시작한다.

01LLMwiki란

원본(raw)과 AI가 구조화한 위키(wiki), 팀 공유처를 분리한 문서 관리 패턴이다. 핵심은 폴더가 아니라 AI에게 주는 규칙 — 그래서 클론할 고정 템플릿이 아니라, 경로만 주면 자라난다.

023레이어 구조

읽기 전용

raw/

원본 기획서·문서·데이터. AI가 절대 수정하지 않는다. 출처 추적용.

진실의 원본

wiki/

Claude가 구조화해 관리. 사람과 AI가 같은 문서를 본다.

선택

공유처

Confluence·Notion 등 팀 공유. wiki에서 동기화.

03핵심 동작 4개

이 위키는 네 명령으로 굴러간다. 사람은 경로/소스만 주고, 나머지 절차는 Claude가 규칙대로 한다.

동작 A

초기화 — 최초 1회

상위 폴더 경로를 주면 → Claude가 로컬용 CLAUDE.md + wiki/ 기초 트리를 만든다.

동작 B

누적 — 반복

소스(Confluence 링크 등)만 던지면 → wiki/{프로젝트}/에 정리해 쌓고 index·log를 갱신한다.

동작 C

조회

"프로젝트 목록 보여줘" → 쌓인 프로젝트를 나열. 골라서 확인 요청.

동작 D · 심화

발행

"컨플루언스에 올려줘 / GitHub에 push해줘" → 팀 공유처로 내보낸다. 연동된 사람만.

성숙도 사다리 — ① 로컬 위키(누구나, 오늘 실습) → ② GitHub 백업(git 있으면) → ③ Confluence 발행(MCP 연동 시). 아래로 갈수록 강력하지만 주의도 커진다.

04실습 — 0. 준비물

기본 (1~4단계, 누구나)

Claude Code 설치 위키화할 소스 1개 (Confluence URL·파일·링크)

심화 (5·6단계, 선택)

git + GitHub 인증 → 5단계 Atlassian(Confluence) MCP 연결 → 6단계

Claude Code가 없다면: docs.claude.com/claude-code · 심화 연동이 없으면 1~4단계만 해도 완결됩니다.

1. 초기화 — 경로만 주기

CLAUDE.md가 2개로 나뉜다 — 아래 URL의 공유용(레시피)을 Claude가 읽고, 내 폴더에 정보가 채워진 로컬용을 새로 만든다. 그 로컬용이 이후 매 세션 자동으로 읽히는 작동 규칙이 된다.
1

아무 빈 폴더나 정하고, Claude Code에 경로를 주며 아래를 붙여넣는다. (클론·다운로드 불필요, 업무/팀 입력 안 해도 됨)

내 LLM wiki를 여기에 만들어줘: ~/work/my-wiki

아래 URL은 "공유용 CLAUDE.md 레시피"야. 이걸 읽고
"로컬용 CLAUDE.md"와 기초 트리(raw/ + wiki/index·log·cross_issues)를 만들어줘.
https://jennyby-ops.github.io/llmwiki-template/CLAUDE.md

→ Claude가 로컬용 CLAUDE.md + 기초 폴더를 만들고 log.md에 INIT를 기록한다. 프로젝트 폴더는 아직 안 만든다 — 소스가 들어올 때 생긴다. 이후 같은 폴더를 열면 로컬용이 자동으로 읽혀, CLAUDE.md를 언급하지 않아도 규칙대로 작동한다.
※ 팀과 공유할 거면 관리자: 이름 / 팀: 팀명 한 줄만 덧붙이면 메타에 적힌다.

2. 누적 — 소스만 던지기

2

위키화할 소스를 던진다. 위키 페이지든, Figma 링크든, 첨부파일이든. 한 줄이면 된다.

이 소스 wiki화해줘:
https://wiki.daumkakao.com/pages/viewpage.action?pageId=000000

→ Claude가 페이지를 읽고 wiki/{프로젝트}/ 4파일을 생성, 출처 URL을 명시하고 대시보드·이력을 갱신한다. 6섹션·다이어그램 분리·index/log 갱신은 CLAUDE.md가 자동 처리라 매번 지시할 필요가 없다. 프로젝트명도 Claude가 판단(애매하면 물어봄) — 직접 정하려면 프로젝트명은 ○○으로만 덧붙이면 된다.

3. 다음 세션부터

3

새 세션에서 한 마디만:

이 프로젝트 파악해줘

Claude가 CLAUDE.md → index.md → log.md → cross_issues.md 순으로 읽고 지난 맥락 위에서 이어간다. 다시 설명할 필요가 없다.

4. 계속 쌓고, 골라 보기

4

그 다음은 동작 B의 반복이다. 소스를 줄 때마다 프로젝트가 쌓인다.

소스 던지기 → wiki화 → index·log 갱신 → (반복)

쌓인 게 많아지면 목록으로 훑고 골라 본다:

프로젝트 목록 보여줘

→ Claude가 index.md 기준으로 프로젝트·버전·상태를 표로 나열한다. 그중 하나를 고르면 해당 overview를 읽어 요약해준다. 여러 프로젝트에 걸친 미결사항은 cross_issues.md로 통합 관리된다.

여기까지가 "누구나 / 오늘 실습" 범위다. 아래 5·6은 연동이 돼 있는 사람만 따라하는 심화(동작 D)다. 실제 GitHub·Confluence에 쓰므로 주의해서 진행한다.

5. GitHub 백업 · 심화

5

전제: 로컬에 git + GitHub 인증(gh 또는 SSH). 위키 폴더를 버전관리·백업·공유한다.

이 위키를 GitHub에 올려줘. 내부 기획 내용이 있으니 private 레포로.
⚠️ private가 기본 — 위키엔 내부 기획이 섞이기 쉽다. public 전환은 "공개해도 되는 내용만 있는지" 직접 확인한 뒤에만. .env·원본 기획서 등 민감 파일은 .gitignore로 빼는지 Claude가 점검한다.

6. Confluence 발행 · 심화

6

전제: Claude Code에 Atlassian(Confluence) MCP가 연결돼 있어야 한다. 안 돼 있으면 이 단계는 건너뛴다.

준비 — 내 개인 공간을 위키 트리로 삼기 (최초 1회)

본인 개인 공간에 프로젝트들을 담을 부모 페이지 하나를 두고, 거기로 발행한다. Claude에게 알려줄 값은 딱 둘: space key와 부모 page ID.

① 개인 공간 들어가기 — 우측 상단 프로필 클릭 → 개인 공간.

② 부모 페이지 만들기 — 개인 공간 홈에서 만들기 버튼으로 카탈로그용 페이지를 하나 만든다 (예: 내 LLMwiki 또는 PRD 모음). 앞으로 모든 프로젝트가 이 아래에 자식 페이지로 쌓인다.

③ URL에서 두 값 뽑기 — 그 부모 페이지의 주소창을 보면 둘 다 들어있다:

wiki.daumkakao.com/spaces/~your.id/pages/1234567890/내-LLMwiki
space key = ~your.id  (개인 공간은 ~ + 본인 아이디. /spaces/ 바로 다음)
부모 page ID = 1234567890  (/pages/ 바로 다음 숫자)

④ Claude에게 한 번 알려주고 기록 — 이후엔 Claude가 CLAUDE.md에 저장해 재사용한다:

내 컨플 발행 설정 기록해줘.
space key: ~your.id / 부모 page ID: 1234567890 (내 개인 공간의 'PRD 모음')
앞으로 발행은 이 페이지 아래에 자식으로 만들어줘.

발행 — 등록 / 갱신

등록 — 프로젝트를 그 부모 아래 페이지로 올린다:

{프로젝트} 를 내 컨플루언스에 페이지로 올려줘.

갱신 — 내용이 바뀌면 다시 올린다 (Claude가 version+1로 충돌 없이 갱신):

{프로젝트} 컨플 페이지 최신 내용으로 갱신해줘.
⚠️ 실제 위키에 쓰는 작업이다. ① 기본은 개인 공간 — 팀 스페이스는 명시 지시할 때만. ② 삭제·덮어쓰기는 Claude가 대상 페이지를 보여주고 확인받은 뒤에만 실행. ③ 생성된 페이지 ID는 CLAUDE.md의 "공유처 페이지 ID 매핑"에 자동 기록돼, 다음부터 같은 페이지를 안전하게 갱신한다.

### 동작 A — 초기화 (최초 1회) ※ 로컬용엔 빠짐
경로를 주면: 로컬용 CLAUDE.md 생성 + raw/ + wiki/(index·log·cross_issues) 생성 + [INIT] 기록.
업무/팀은 묻지 않음(여러 프로젝트를 담는 그릇). 프로젝트 폴더는 누적 때 생성.

### 동작 B — 누적 (반복)
소스만 주면: 소스 읽기 → 프로젝트 판단(애매하면 확인) → overview 6섹션 작성,
다이어그램은 flows.md → 출처 명시 → index.md·log.md 갱신 → 교차 미결은 cross_issues.md.

### 동작 C — 조회
"프로젝트 목록 보여줘" → index.md 기준 프로젝트·버전·상태 나열 → 고르면 overview 요약.

### 동작 D — 발행 (심화) ※ MCP/git 연동 시
Confluence: 개인 스페이스 기본 · 수정은 version+1 · 삭제/덮어쓰기 전 확인 · pageId 매핑 기록.
GitHub: 내부 내용이면 private 레포 · 민감파일 .gitignore 점검.

09FAQ

레포를 클론해야 하나요?

아니요. 1번 초기화 프롬프트만 붙여넣으면 Claude가 필요한 파일을 직접 만듭니다. 이 레포는 가이드 호스팅 + 참고용입니다.

Confluence 연동이 안 돼 있으면?

누적(동작 B)은 로컬 파일·링크로 대체하면 되고, 발행(동작 D-Confluence)은 그냥 건너뛰면 됩니다. 1~5단계(로컬 위키 + GitHub)만으로도 완결됩니다. Confluence 발행은 Atlassian MCP가 연결된 사람만 6단계로 진행하세요.

실습장에서 Confluence에 잘못 쓰면 어쩌죠?

그래서 기본이 개인 스페이스이고, 삭제·덮어쓰기는 Claude가 대상을 보여주고 확인받은 뒤에만 실행합니다. 팀 스페이스는 명시적으로 지시할 때만 건드립니다. 발표 시연도 본인 개인 스페이스에서 하시길 권합니다.

CLAUDE.md가 너무 길어지면?

안 쓰는 규칙(예: 공유처 동기화)을 지우세요. 짧고 핵심만 있는 CLAUDE.md가 가장 잘 작동합니다.
