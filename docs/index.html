Claude Code를 PM 업무의 협업자로 쓰는 법

경로만 주면 자라나는 위키 — LLMwiki 워크플로우

물류플랫폼팀 AI 스터디 · 발표자 jenny.by · 클론 없이, 프롬프트 2개로 내 Mac에 로컬 LLMwiki를 만들고 키웁니다.

00왜 필요한가

AI에게 업무를 시켜본 사람은 안다. 매번 처음부터 설명해야 한다. "우리 팀은 미들마일이고, 이 기획서는 v1.6이고…" 세션이 끝나면 그 맥락은 전부 사라진다.

컨텍스트 소실 문제 — LLM은 대화가 끝나면 기억하지 못한다. 그래서 매 세션 같은 설명을 반복하고, 결과물의 포맷·깊이가 들쭉날쭉해진다.

LLMwiki는 이걸 파일로 푼다. 업무 컨텍스트와 작동 규칙을 CLAUDE.md에 한 번 적어두면, Claude Code가 매 세션 그 파일부터 읽고 시작한다.

01LLMwiki란

원본(raw)과 AI가 구조화한 위키(= 로컬 LLMwiki), 팀 공유처(Confluence)를 분리한 문서 관리 패턴이다. 핵심은 폴더가 아니라 AI에게 주는 규칙 — 그래서 클론할 고정 템플릿이 아니라, 경로만 주면 자라난다.

📍 "위키"가 두 개라 헷갈리지 않게
이 가이드엔 위키가 둘 나옵니다. 구분해서 표기해요:
• 로컬 LLMwiki — 내 Mac의 ~/Documents/myLLMwiki 폴더. Claude가 만들고 관리하는 내 작업용 위키.
• Confluence(공용 위키) — 회사가 함께 쓰는 위키. 6단계에서 여기로 발행한다.
흐름: 회사 Confluence·자료 → 내 로컬 LLMwiki에 정리 → (선택) 다시 Confluence로 발행.

023레이어 구조

읽기 전용

raw/

원본 기획서·문서·데이터. AI가 절대 수정하지 않는다. 출처 추적용.

진실의 원본 · 로컬 LLMwiki

wiki/

내 Mac에서 Claude가 구조화해 관리. 사람과 AI가 같은 문서를 본다.

선택 · 공용

Confluence

회사 공용 위키. 로컬 LLMwiki에서 여기로 발행.

03핵심 동작 4개

이 로컬 LLMwiki는 네 명령으로 굴러간다. 사람은 경로/소스만 주고, 나머지 절차는 Claude가 규칙대로 한다.

동작 A

초기화 — 최초 1회

상위 폴더 경로를 주면 → Claude가 로컬용 CLAUDE.md + wiki/ 기초 트리를 만든다.

동작 B

누적 — 반복

소스 + 프로젝트명을 던지면 → wiki/{프로젝트}/에 정리해 쌓고 index·log를 갱신한다. (이름 안 주면 목록 보여주고 물어봄)

동작 C

조회

"프로젝트 목록 보여줘" → 쌓인 프로젝트를 나열. 골라서 확인 요청.

동작 D · 심화

발행

"사내 km-deploy에 배포해줘 / 컨플루언스에 올려줘" → 공유처로 내보낸다. 연동된 사람만.

성숙도 사다리 — ① 로컬 LLMwiki(누구나, 오늘 실습) → ② 사내 배포(km-deploy로 미리보기 공유) → ③ Confluence(공용 위키) 발행(MCP 연동 시). 아래로 갈수록 강력하지만 주의도 커진다.

04실습 — 0. 준비물

기본 (1~4단계, 누구나)

Claude Code 설치 위키화할 소스 1개 (Confluence URL·파일·링크)

심화 (5·6단계, 선택)

km-deploy PAT 토큰 → 5단계 Atlassian(Confluence) MCP 연결 → 6단계

Claude Code가 없다면: docs.claude.com/claude-code · 심화 연동이 없으면 1~4단계만 해도 완결됩니다.

1. 초기화 — 경로만 주기

CLAUDE.md가 2개로 나뉜다 — 아래 URL의 공유용(레시피)을 Claude가 읽고, 내 폴더에 정보가 채워진 로컬용을 새로 만든다. 그 로컬용이 이후 매 세션 자동으로 읽히는 작동 규칙이 된다.
1

Claude Code에 아래를 그대로 붙여넣는다. (클론·다운로드 불필요)

내 LLM wiki를 여기에 만들어줘: ~/Documents/myLLMwiki

아래 URL은 "공유용 CLAUDE.md 레시피"야. 이걸 읽고
"로컬용 CLAUDE.md"와 기초 트리(raw/ + wiki/index·log·cross_issues)를 만들어줘.
https://jennyby-ops.github.io/llmwiki-template/CLAUDE.md
  1. 경로 ~/Documents/myLLMwiki 는 원하는 곳으로 바꿔도 된다. 그냥 그대로 둬도 OK.
  2. 이걸 보내면 Claude가 그 폴더에 로컬 LLMwiki의 뼈대(빈 폴더와 안내 파일)를 자동으로 만들어준다. 아직 내용물은 없고, 다음 단계에서 자료를 넣으면 채워진다. 한 번 만들면 다음부터는 폴더만 열어도 Claude가 알아서 규칙을 읽는다.

※ 팀과 공유할 거면 위 프롬프트에 관리자: 이름 / 팀: 팀명 한 줄만 덧붙이면 된다.

➕ 고도화 (선택) — 내 문서 스타일 학습 · 추천

＋

사람마다 기획서 쓰는 방식이 다르다. 내 기존 기획서 2~3개를 참고시키면, Claude가 내 섹션 구성·문장 톤·표 형식·제목 규칙을 학습해 로컬 CLAUDE.md에 "내 표준"으로 반영한다. 이후 정리가 처음부터 내 스타일로 나온다.

내 기존 기획서 2~3개 참고해서 내 문서 스타일 익혀줘:
(Confluence 링크)
(Figma 링크)
(또 다른 기획서 링크·파일)
  1. Confluence·Figma·첨부파일 아무거나 된다. 완성도 높은 대표 문서일수록 학습이 정확하다 (2~3개 권장).
  2. Claude가 학습한 스타일을 요약해 보여주고 확인받은 뒤 CLAUDE.md에 "내 표준"으로 저장한다.
  3. 안 해도 된다 — 그럼 표준 6섹션 포맷으로 정리된다. 나중에 언제든 학습시켜도 OK.

2. 누적 — 소스 + 프로젝트명 던지기

2

위키화할 소스를 어느 프로젝트에 넣을지(프로젝트명)와 함께 던진다.

이 소스 wiki화해줘. 프로젝트명은 OOO이야:
https://wiki.daumkakao.com/pages/viewpage.action?pageId=000000
  1. Claude가 읽을 수 있는 자원은 거의 다 된다. Confluence(공용 위키) 링크, Figma 링크, 첨부파일(PDF·엑셀 등), 로컬 파일 경로, 그냥 붙여넣은 텍스트까지.
  2. 프로젝트명을 같이 주면 그 칸에 정리한다 — 같은 이름의 기존 프로젝트면 거기에 누적, 새 이름이면 새 프로젝트로 만든다.
  3. 프로젝트명을 안 주면 Claude가 바로 정리하지 않고 현재 프로젝트 목록을 보여주며 "어디에 넣을까? 새 거면 이름 정해줘"를 먼저 물어본다. (그래야 엉뚱한 칸에 안 쌓인다)
  4. 정해지면 그 내용을 보기 좋게 정리해 로컬 LLMwiki에 쌓고 변경 기록(log)을 남긴다. 소스를 줄 때마다 로컬 LLMwiki가 점점 자란다.

3. 다음 세션부터

3

새 세션에선 폴더만 열고 한 마디면 된다. 먼저 목록으로 프로젝트명을 확인한 뒤, 그 이름으로 명령한다:

프로젝트 목록 보여줘            ← 먼저 이걸로 프로젝트명 확인
이 프로젝트 파악해줘: (프로젝트명)
방금 회의 내용 정리해서 내 LLMwiki에 반영해줘: (프로젝트명)
(프로젝트명)에 이 내용 업데이트해줘: (소스/텍스트)
(프로젝트명) 지금 미결사항 뭐 남았어?
  1. "프로젝트 목록 보여줘"로 먼저 프로젝트명을 확인하고, 그 이름을 넣어 명령하면 정확히 그 칸에 정리된다.
  2. Claude가 매번 CLAUDE.md → index → log를 먼저 읽어서, 지난 세션 맥락 위에서 이어간다. 처음부터 설명할 필요가 없다.
  3. 프로젝트명 없이 회의록·자료를 던지면, Claude가 목록을 보여주며 어느 프로젝트인지 먼저 물어본다.

4. 계속 쌓고, 골라 보기

4

그 다음은 동작 B의 반복이다. 소스를 줄 때마다 프로젝트가 쌓인다.

소스 던지기 → wiki화 → index·log 갱신 → (반복)

쌓인 게 많아지면 목록으로 훑고 골라 본다:

프로젝트 목록 보여줘

→ Claude가 index.md 기준으로 프로젝트·버전·상태를 표로 나열한다. 그중 하나를 고르면 해당 overview를 읽어 요약해준다. 여러 프로젝트에 걸친 미결사항은 cross_issues.md로 통합 관리된다.

여기까지가 "누구나 / 오늘 실습" 범위다. 아래 5·6은 연동이 돼 있는 사람만 따라하는 심화(동작 D)다. 실제 사내 서버·Confluence에 쓰므로 주의해서 진행한다.

5. 사내 서버 배포 (km-deploy) · 심화

5

전제: 사내 배포 서비스 km-deploy PAT 토큰. 로컬 LLMwiki를 정적 페이지로 만들어 사내에서만 링크로 공유한다.

내 LLMwiki를 사내 km-deploy에 배포해줘.
  1. 외부(GitHub) 대신 사내 서버에만 올라가므로 내부 기획 내용도 안전하다. (보안상 외부 공개 금지)
  2. Claude가 PAT 토큰으로 파일을 업로드하고 배포 링크를 만들어준다. 토큰이 없으면 이 단계는 건너뛰고 1~4단계 로컬 LLMwiki로 충분하다.

6. Confluence 발행 · 심화

6

전제: Claude Code에 Atlassian(Confluence) MCP가 연결돼 있어야 한다. 안 돼 있으면 이 단계는 건너뛴다.

준비 — Confluence 개인 공간을 발행 트리로 삼기 (최초 1회)

회사 Confluence 개인 공간에 프로젝트들을 담을 부모 페이지 하나를 두고, 로컬 LLMwiki를 거기로 발행한다. Claude에게 알려줄 값은 딱 둘: space key와 부모 page ID.

① 개인 공간 들어가기 — 우측 상단 프로필 클릭 → 개인 공간.

② 부모 페이지 만들기 — 개인 공간 홈에서 만들기 버튼으로 카탈로그용 페이지를 하나 만든다 (예: 내 LLMwiki 또는 PRD 모음). 앞으로 모든 프로젝트가 이 아래에 자식 페이지로 쌓인다.

③ URL에서 두 값 뽑기 — 그 부모 페이지의 주소창을 보면 둘 다 들어있다:

wiki.daumkakao.com/spaces/~your.id/pages/1234567890/내-LLMwiki
space key = ~your.id  (개인 공간은 ~ + 본인 아이디. /spaces/ 바로 다음)
부모 page ID = 1234567890  (/pages/ 바로 다음 숫자)

④ Claude에게 한 번 알려주고 기록 — 이후엔 Claude가 CLAUDE.md에 저장해 재사용한다:

내 컨플 발행 설정 기록해줘.
space key: ~your.id / 부모 page ID: 1234567890 (내 개인 공간의 'PRD 모음')
앞으로 발행은 이 페이지 아래에 자식으로 만들어줘.

발행 — 로컬 LLMwiki → Confluence (프로젝트별 버전관리)

등록 — 프로젝트(메뉴)를 컨플에 올린다. Claude가 그 부모 아래에 프로젝트 인덱스 페이지 + 버전 하위 페이지(예: v1.0)를 만든다:

{프로젝트} 를 내 컨플루언스에 올려줘.

갱신 — 내용이 바뀌면 같은 버전 페이지를 갱신한다 (version+1로 충돌 없이):

{프로젝트} 컨플 페이지 최신 내용으로 갱신해줘.

버전 올리기 — 큰 변경이면 새 버전 페이지를 만들고 이전 버전은 그대로 보존한다:

{프로젝트} 를 v2로 새 버전 페이지 만들어줘. 이전 버전은 남겨두고.
  1. 프로젝트(메뉴)마다 버전이 따로 쌓인다. 인덱스 페이지에 v1.0 → v1.1 → v2 식으로 버전 이력이 남고, 옛 버전 페이지도 지우지 않고 보존한다.
  2. 변경 전 Claude가 "챗에 변경 요약 → 로컬 LLMwiki에 반영할까요? Confluence에도 올릴까요?"를 먼저 물어본다. 삭제·덮어쓰기는 대상 페이지를 보여주고 동의받은 뒤에만 실행한다.
  3. 기본은 본인 개인 공간. 팀 스페이스는 명시적으로 지시할 때만. 생성된 페이지 ID는 자동으로 기록돼 다음부터 안전하게 갱신된다.

07CLAUDE.md 전문

로컬 LLMwiki의 모든 동작은 이 파일 하나에서 나온다. 원문 열기 →

CLAUDE.md 펼쳐보기 (동작 3개 핵심) 
### 동작 A — 초기화 (최초 1회) ※ 로컬용엔 빠짐
경로를 주면: 로컬용 CLAUDE.md 생성 + raw/ + wiki/(index·log·cross_issues) 생성 + [INIT] 기록.
업무/팀은 묻지 않음(여러 프로젝트를 담는 그릇). 프로젝트 폴더는 누적 때 생성.

### 동작 B — 누적 (반복)
소스 + 프로젝트명을 주면: 소스 읽기 → (이름 없으면 목록 보여주고 어느 프로젝트인지 먼저 확인) →
overview 6섹션 작성, 다이어그램은 flows.md → 출처 명시 → index.md·log.md 갱신 → 교차 미결은 cross_issues.md.

### 동작 C — 조회
"프로젝트 목록 보여줘" → index.md 기준 프로젝트·버전·상태 나열 → 고르면 overview 요약.

### 동작 D — 발행 (심화) ※ 연동 시
변경은 항상 챗에 먼저 정리 → "로컬 위키 / Confluence 중 어디에 반영할까요?" 확인 후 진행.
사내 배포: km-deploy(PAT)로 정적 배포 — 외부 금지, 사내만.

09FAQ

레포를 클론해야 하나요?

아니요. 1번 초기화 프롬프트만 붙여넣으면 Claude가 필요한 파일을 직접 만듭니다. 이 레포는 가이드 호스팅 + 참고용입니다.

Confluence 연동이 안 돼 있으면?

누적(동작 B)은 로컬 파일·링크로 대체하면 되고, 발행(동작 D-Confluence)은 그냥 건너뛰면 됩니다. 1~4단계 로컬 LLMwiki만으로도 완결됩니다. Confluence 발행은 Atlassian MCP가 연결된 사람만 6단계로 진행하세요.

실습장에서 Confluence에 잘못 쓰면 어쩌죠?

그래서 기본이 개인 스페이스이고, 삭제·덮어쓰기는 Claude가 대상을 보여주고 확인받은 뒤에만 실행합니다. 팀 스페이스는 명시적으로 지시할 때만 건드립니다. 발표 시연도 본인 개인 스페이스에서 하시길 권합니다.

CLAUDE.md가 너무 길어지면?

안 쓰는 규칙(예: 공유처 동기화)을 지우세요. 짧고 핵심만 있는 CLAUDE.md가 가장 잘 작동합니다.
