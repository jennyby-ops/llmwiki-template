# CLAUDE.md — LLMwiki 초기화 레시피 (공유용)

> ⚠️ **이 파일은 "공유용 레시피"다.** github.io에 올라가 있고, 사용자는 초기화 프롬프트에서
> 이 URL을 참조한다. 이 파일을 그대로 작업 폴더에 두고 쓰는 게 아니다.
>
> 이 파일을 읽은 Claude는 **동작 A(초기화)**를 수행해, 사용자의 작업 폴더에
> **"로컬용 CLAUDE.md"를 새로 생성**한다. 그 로컬용 파일이 앞으로 매 세션 자동으로 읽히는
> 실제 작동 규칙이 된다.
>
> 공유 레시피 관리: jenny.by · 물류플랫폼팀 AI 스터디

---

## ⭐ 공통 규칙 — 변경 반영 전 확인 (모든 동작에 우선 적용)

위키/공유처에 무언가를 쓰기 전에 Claude는 **항상**:
1. **변경 내용을 챗에 먼저 요약**한다 (무엇을 추가/수정/삭제할지).
2. 그다음 **"어디에 반영할까요?"를 물어본다**:
   - ① **로컬 LLMwiki에 반영**할까요?
   - ② **Confluence(WIKI)에도 발행**할까요?
3. 사용자가 고른 곳에만 적용한다. (둘 다 / 로컬만 / 보류 선택 가능)

> 단순 조회(동작 C)나 사용자가 "바로 반영해"라고 명시한 경우엔 이 확인을 생략할 수 있다.

---

## ⭐ 동작 A — 초기화 (사용자당 최초 1회)

> 트리거 예: "내 LLM wiki를 여기에 만들어줘: `/path`" (+ 이 공유용 URL 참조)

사용자가 **상위 폴더 경로**를 주면 Claude는:

1. **로컬용 CLAUDE.md 생성** — 그 경로에 `CLAUDE.md`를 새로 쓴다. 이 공유용을 기반으로 하되:
   - **이 "동작 A(초기화)" 섹션은 뺀다.** (초기화는 1회용이므로 로컬용엔 불필요)
   - "동작 B(누적)" · "동작 C(조회)" · "변하지 않는 규칙" · "overview 6섹션 포맷"을 남긴다.
   - 맨 위에 "이 파일은 Claude Code가 이 폴더를 열 때 자동으로 읽는 작동 규칙"이라고 명시.
2. `raw/` 폴더 생성 (`.gitkeep`).
3. `wiki/index.md`, `wiki/log.md`, `wiki/cross_issues.md` 생성 (빈 골격).
4. `wiki/log.md`에 `[날짜] [INIT] wiki 초기화 — 로컬 CLAUDE.md + 기초 트리 생성` 기록.

> ℹ️ **업무/팀은 묻지 않는다.** 이 위키는 한 가지 업무가 아니라 **여러 프로젝트를 담는 그릇**이다.
> 단, 사용자가 "관리자/팀"을 함께 주면(= 같은 팀과 공유할 목적) 로컬용 상단 메타에 적는다. 안 주면 비워둔다.
>
> ⚠️ 초기화 시 프로젝트(`{프로젝트명}/`) 폴더는 **만들지 않는다.** 프로젝트는 동작 B로 소스가 들어올 때 생성된다.
>
> ✅ 초기화 후: 같은 폴더에서 Claude Code를 열면 **로컬용 CLAUDE.md가 자동으로 읽히므로**,
> 앞으로는 규칙(6섹션·flows·index/log 등)을 **매번 말하지 않아도** 그대로 작동한다.

## ⭐ 동작 B — 누적 (반복) — *로컬용에 포함*

> 트리거 예: "이 소스 wiki화해줘: {Confluence URL}" — **이 한 줄이면 충분하다.**
> 6섹션·flows 분리·index/log 갱신은 사용자가 지시하지 않아도 아래 절차대로 **자동 수행**한다.

사용자가 **소스(+선택: 프로젝트명)**를 주면 Claude는:

1. **소스를 읽는다.**
   - Confluence URL → Atlassian MCP `confluence_get_page` (pageId 또는 URL)
   - 첨부파일/로컬 경로 → 먼저 `raw/`에 보관 후 읽기
2. **프로젝트를 판단한다.**
   - 사용자가 프로젝트명을 지정했으면 그대로.
   - 기존 프로젝트에 속하면 → 해당 `wiki/{프로젝트명}/` 문서에 **append/갱신** (덮어쓰지 말 것).
   - 신규면 → `wiki/{프로젝트명}/` 폴더 + 4파일(overview·issues·meetings·flows) 생성.
   - **애매하면 신규/기존 여부를 사용자에게 먼저 확인** (잘못된 분류 방지).
3. **overview.md를 6섹션 포맷으로 작성.** 다이어그램 코드는 `flows.md`에만, 본문엔 경로 참조만.
4. **원본 출처를 기록** — overview.md 상단에 소스 URL/경로 명시.
5. **index.md 대시보드 행 추가/갱신 + log.md에 `[ADD]`/`[UPDATE]` 기록.**
6. 2개 이상 프로젝트에 걸친 미결은 → `cross_issues.md`로 승격.

> 매번 이 절차를 거치면 세션을 반복할수록 위키가 **누적**된다.

## ⭐ 동작 C — 조회 / 목록 — *로컬용에 포함*

> 트리거 예: "프로젝트 목록 보여줘" / "지금까지 정리된 거 뭐 있어?" / "○○ 프로젝트 확인해줘"

- **목록 요청** → `wiki/index.md` 대시보드를 기준으로 **프로젝트명·버전·상태·마지막 업데이트**를 표로 나열한다.
- 사용자가 그중 하나를 고르면 → 해당 `wiki/{프로젝트명}/overview.md`(필요 시 issues·flows)를 읽어 요약해 보여준다.
- index.md와 실제 폴더가 어긋나 있으면 폴더 기준으로 알려주고 index.md를 보정한다.

## ⭐ 동작 D — 발행 (심화/선택) — *로컬용에 포함*

> 트리거 예: "이 프로젝트 컨플루언스에 올려줘 / 갱신해줘" · "사내 km-deploy에 배포해줘"
> **전제**: Confluence는 Atlassian MCP 연결, 사내 배포는 km-deploy PAT 토큰이 있어야 한다. 안 돼 있으면 동작 D는 건너뛴다.
> ⚠️ 발행 전에는 위 "공통 규칙 — 변경 반영 전 확인"에 따라 챗 요약 + 반영처 확인을 먼저 한다.

**발행 설정 (최초 1회 확인 후 아래 "공유처 페이지 ID 매핑" 상단에 기록)**
- **space key**: 발행 대상 스페이스. 개인 공간은 `~` + 본인 아이디 (예: `~jenny.by`). Confluence URL `…/spaces/{spaceKey}/…`에서 추출.
- **root pageId**: 모든 프로젝트 페이지를 담을 부모 페이지 ID. 부모 페이지 URL `…/pages/{pageId}/…`에서 추출 (예: `1234567890`).
- 설정이 없으면 발행 전에 사용자에게 두 값을 물어 확정하고 기록한다.

### D-1. Confluence 등록 / 수정 / 버전관리 (개인 위키 → 컨플루언스, 프로젝트=메뉴별 버전관리)
구조: **root 부모 페이지 → 프로젝트(메뉴) 인덱스 페이지 → 버전 하위 페이지(v1.0, v1.1 …)**.
1. **등록** — `wiki/{프로젝트}/overview.md`를 컨플 페이지로 만든다. **space key의 root pageId 아래**에:
   - 프로젝트별 **인덱스 페이지** 1개 (버전 이력·링크 모음) + 그 아래 **버전 하위 페이지**(예: `{프로젝트} v1.0`).
   - root에 전체 **카탈로그 인덱스**(예: "PRD 인덱스")가 없으면 함께 만든다.
   - 생성된 모든 `pageId`를 아래 "공유처 페이지 ID 매핑"에 기록한다.
2. **수정(같은 버전)** — 반드시 **GET 현재 version → PUT version+1** 패턴으로 갱신한다 (409 충돌 방지).
3. **버전 올리기** — 큰 변경은 **새 버전 하위 페이지를 생성**하고 **이전 버전 페이지는 보존**(삭제 금지). 프로젝트 인덱스에 버전 이력을 추가하고, 카탈로그 인덱스 링크를 최신 버전으로 갱신한다.
4. **삭제 / 덮어쓰기** — ⚠️ 실행 전 **사용자에게 반드시 확인**한다. 대상 페이지의 **제목·ID·스페이스**를 보여주고 동의를 받은 뒤에만.

> ⚠️ 실제 위키에 쓰는 작업이다. 기본은 **개인 스페이스**. **팀 스페이스 발행은 사용자가 명시적으로 지시할 때만.** 각 메뉴(프로젝트)는 버전이 독립적으로 쌓인다.

### D-2. 사내 서버 배포 (km-deploy) — 외부 금지
1. 위키를 정적 페이지로 만들어 사내 **km-deploy** (`https://km-deploy.onkakao.net/`)에 PAT 토큰으로 배포한다 (업로드 → 프로젝트 생성/덮어쓰기). `km-deploy` 스킬이 있으면 그것을 사용한다.
2. ⚠️ **보안상 외부(GitHub 등 사외) 배포·공개는 하지 않는다.** 내부 기획 내용은 사내 서버에만 둔다.
3. 토큰/연동이 없으면 이 단계는 건너뛴다 (로컬 위키로 충분).

> 발행 후 `log.md`에 `[PUBLISH]` 이력을 남기고, 필요 시 index.md의 공유처 링크를 갱신한다.

## 이 위키는 무엇인가 — *로컬용에 포함*

{이름/조직}의 여러 프로젝트 기획 소스를 한 곳에 누적·구조화하는 개인 위키.
한 프로젝트 = `wiki/{프로젝트명}/` 폴더 하나. 시간이 지나며 프로젝트가 쌓인다.

**3레이어**: raw(원본) → wiki(AI가 구조화, 진실의 원본) → {공유처, 선택}

- (선택) 관리자: {담당자} — 팀과 공유할 때만
- (선택) 팀: {소속/팀} — 팀과 공유할 때만
- (선택) 공유처: {Confluence/Notion 등} `{URL}` — 안 쓰면 삭제

## 변하지 않는 규칙 — *로컬용에 포함*

### 1. raw/ 는 읽기 전용
원본은 절대 수정하지 않는다. 참조만. (출처 추적 + 원본 보존)

### 2. wiki/ 변경 시 항상 index.md + log.md 갱신
어떤 wiki 파일을 수정해도 `index.md` 현황 + `log.md` 이력(`[날짜] [유형] [대상] — 내용`)을 반드시 갱신.

### 3. Mermaid — 본문에 코드 직접 삽입 금지
다이어그램 코드는 `flows.md`에만. overview.md엔 참조만:
```
[mermaid 플로우 직접입력 - wiki/{프로젝트명}/flows.md → Flow {N}]
```

### 4. cross_issues.md — 세션 간 공유 미결사항
세션 시작 필독 순서: `index.md → log.md → cross_issues.md → {내프로젝트}/`.
2개 이상 프로젝트에 걸친 이슈는 X번으로 통합 관리, 각 프로젝트 issues.md엔 "→ cross_issues.md X번" 메모만.

### 5. 버전업 시 누락 방지 체크 (필수)
재구조화하며 버전업할 때 이전 버전의 **모든 항목**을 나열 → ✅반영 / 🔄변형흡수 / ❌누락 매핑 → ❌는 "의도 제외인지 누락인지" 사용자에게 **반드시 확인**.
> 추상화는 디테일을 잃기 쉽다. "더 깔끔해졌다"는 곧 "무언가 빠졌다"일 수 있다.

### 6. 발행은 동작 D 규칙을 따른다
발행은 위 **동작 D**의 절차(반영처 확인 · 개인 스페이스 기본 · 메뉴별 version+1/버전 보존 · 삭제/덮어쓰기 전 확인 · **외부 배포 금지, 사내 km-deploy만**)를 반드시 지킨다. 발행을 안 쓰면 동작 D와 아래 매핑 표는 삭제해도 된다.

## overview.md 6섹션 포맷 — *로컬용에 포함*

```
1. 개요 및 배경   — 변경 이유 + 핵심 요약
2. 개념/구조      — 관계 트리 + 핵심 개념 테이블
3. Flow          — [mermaid 플로우 직접입력 - wiki/{프로젝트명}/flows.md → Flow {N}]
4. 변경 범위      — AS-IS | TO-BE 테이블
5. 상세 스펙      — 항목 | 필수 | 설명 테이블
6. 미결사항       — # | 항목 | 담당 | 상태 테이블
```
스타일: 미결/보류 삭제 금지(인라인 메모 유지) · 이번 범위 vs 추후 고도화 분리 · NEW 태그로 신규/변경/삭제 구분.
